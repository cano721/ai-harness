# AI Harness - 개요

## AI Harness란?

AI Harness는 Claude Code 플러그인 형태의 **팀별 AI 에이전트 셋업 시스템**이다.

플러그인을 설치하고 `/harness-init`을 실행하면, 프로젝트를 분석하여 팀에 맞는 보안 Hook, 코드 컨벤션, 스킬을 자동으로 구성한다. 에이전트가 작업할 때 이 설정이 상시 적용되어 안전하고 일관된 코드를 만들도록 돕는다.

## 왜 필요한가

AI 에이전트를 팀에서 사용하면 세 가지 문제가 반복된다.

### 1. 보안 위험
에이전트는 `rm -rf /`, `DROP TABLE`, `force push` 같은 위험 명령을 실행할 수 있고, 시크릿을 코드에 하드코딩하거나 민감 파일을 커밋할 수 있다. 사람이 매번 검토하기엔 속도와 누락의 문제가 있다.

### 2. 일관성 부재
같은 팀에서도 에이전트마다 다른 네이밍, 다른 패턴, 다른 구조로 코드를 생성한다. 컨벤션 문서가 있어도 에이전트가 자동으로 참조하지 않으면 무의미하다.

### 3. 온보딩 비용
새 프로젝트나 새 팀원이 에이전트를 사용하려면 보안 설정, 컨벤션 파악, 외부 도구 연동을 매번 수동으로 해야 한다. 이 과정이 표준화되어 있지 않으면 팀마다 제각각이 된다.

**AI Harness는 이 세 문제를 "한 번 init, 이후 자동 적용"으로 해결한다.**

## 설계 철학

| # | 철학 | 설명 |
|---|------|------|
| 1 | **추천 + 선택** | 하네스가 베스트 프랙티스를 추천하고, 팀이 선택한다. 강요하지 않는다. |
| 2 | **최소 강제** | 필수는 보안 Hook 4개뿐. 나머지 모든 설정은 opt-in이다. |
| 3 | **팀 자율성** | 각 팀이 자기 도메인, 컨벤션, 스킬, Hook을 자유롭게 구성한다. |
| 4 | **셋업 후 빠지기** | init 시 세팅해주고, 이후엔 Claude Code가 알아서 동작한다. 하네스는 개입하지 않는다. |
| 5 | **차단이 아닌 안내** | 위반 시 단순 차단이 아니라 구체적 대안 코드를 제시한다. |

## 어떻게 동작하는가

```
[1단계] 설치          claude plugin install ai-harness
                      → 플러그인이 Claude Code에 등록됨

[2단계] 초기화        /harness-init
                      → [1] 팀 선택 — "어떤 팀 설정을 사용할까요?"
                      → [2] 글로벌 세팅 확인 — 보안 Hook 적용 안내 + 확인
                      → [3] 프로젝트 확인 — 현재 프로젝트 파악 + 확인
                      → [4] 프로젝트 세팅 — 미세팅 항목 표시 + 선택

[3단계] 사용          일반적으로 코딩 작업 수행
                      → Claude Code가 등록된 Hook으로 자동 보안 검증
                      → 에이전트가 컨벤션/컨텍스트맵 참조
                      → 하네스는 개입하지 않음
```

### 설치 후 프로젝트에 생기는 것

```
프로젝트/
├── .ai-harness/
│   ├── config.yaml              # 프로젝트 설정 (팀, 가드레일, 아키텍처)
│   ├── context-map.md           # 에이전트용 프로젝트 지도
│   ├── pending-decisions.yaml   # 미결정 사항 (init 중 보류된 질문)
│   ├── logs/                    # 감사 로그
│   └── teams/{team}/            # 팀별 컨벤션, Hook, 스킬
├── .claude/
│   ├── CLAUDE.md                # 하네스 규칙 주입 (기존 내용 보존)
│   └── settings.json            # Hook 등록 (기존 설정 보존)
```

### 셋업 후 자동으로 동작하는 것

| 시점 | 동작 | Hook |
|------|------|------|
| 도구 사용 전 | 위험 명령 차단 + 대안 안내 | `block-dangerous.sh` |
| 도구 사용 전 | 시크릿 하드코딩 감지 | `secret-scanner.sh` |
| 도구 사용 전 | 아키텍처 경계 위반 검증 | `check-architecture.sh` |
| 도구 사용 후 | 모든 도구 사용 로깅 | `audit-logger.sh` |

## 누가 사용하는가

### Backend 팀
- **init 시**: Java/Spring 감지 → 패키지 구조, API 패턴, DTO 네이밍, 예외 클래스 분석 → 맞춤 컨벤션 생성
- **작업 시**: `/harness-scaffold crud` → 컨벤션 기반 Controller/Service/Repository/Entity 일괄 생성
- **자동 검증**: SQL 인젝션 방지, API 호환성 검사, 엔티티 변경 리뷰

### Frontend 팀
- **init 시**: React/Vue/Next.js 감지 → 컴포넌트 구조, 상태 관리, 스타일링 패턴 분석
- **작업 시**: 컨벤션 기반 컴포넌트 생성
- **자동 검증**: 번들 사이즈 체크, Lighthouse 성능 검사

### DevOps 팀
- **init 시**: Dockerfile, k8s, CI/CD 설정 감지 → 인프라 패턴 분석
- **작업 시**: 배포 체크리스트, 롤백 계획 생성
- **자동 검증**: 인프라 변경 리뷰

### QA 팀
- **init 시**: 테스트 프레임워크, 커버리지 설정 감지
- **작업 시**: 테스트 시나리오 생성, 회귀 테스트 관리

### Planning 팀
- **init 시**: 기존 문서 구조, 이슈 추적기 연동 파악
- **작업 시**: PRD, 유저스토리 작성 (Given-When-Then 수용 기준 포함)

### Design 팀
- **init 시**: 디자인 시스템, 토큰, 컴포넌트 라이브러리 파악
- **작업 시**: 접근성 검사, 반응형 가이드, Figma-to-Code 변환

## 무엇이 아닌가

| 오해 | 실제 |
|------|------|
| AI 에이전트 자체 | 하네스는 에이전트가 아니다. Claude Code 위에서 동작하는 **설정 레이어**다. |
| 코드 생성기 | 하네스가 코드를 생성하지 않는다. 에이전트가 **컨벤션에 맞게** 생성하도록 안내한다. |
| 모니터링 도구 | 실시간 모니터링이 아니라, 도구 사용 시점의 **검증과 로깅**이다. |
| 필수 프레임워크 | 보안 Hook 4개 외에는 모두 선택이다. 팀이 원하지 않으면 **빼도 된다**. |
| 코드 리뷰어 | 사람의 코드 리뷰를 대체하지 않는다. 에이전트의 **기계적 실수를 방지**한다. |

## 사용 가능한 스킬

| 스킬 | 설명 |
|------|------|
| `/harness-init` | 프로젝트 분석 → 컨벤션 생성 → Hook 등록 |
| `/harness-status` | 상태 확인 + 차단 현황 + 진단 + 미결정 사항 |
| `/harness-rules` | 적용 중인 규칙 + 마지막 차단 사유 |
| `/harness-team` | 팀 추가/수정/제거/목록 |
| `/harness-exclude` | 글로벌 하네스에서 프로젝트 제외 |
| `/harness-metrics` | 에이전트 작업 효율 메트릭 분석 + 개선 제안 |
| `/harness-scaffold` | 컨벤션 기반 코드 보일러플레이트 생성 |
