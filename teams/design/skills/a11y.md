# 접근성(A11y) 검증 Skill

## 사용법
/harness a11y [파일 경로 또는 컴포넌트명]

예시: `/harness a11y src/components/ApplicantCard.tsx`
예시: `/harness a11y src/pages/JobPosting/`

## 실행 내용

지정된 파일 또는 디렉토리의 React 컴포넌트를 WCAG 2.1 AA 기준으로 접근성 감사를 수행하고, 문제점과 개선 방안을 제시하라.

### 검사 항목

**1. 명암비 (Color Contrast)**
- 일반 텍스트: 최소 4.5:1 비율
- 큰 텍스트(18pt 이상 또는 14pt Bold): 최소 3:1 비율
- UI 컴포넌트 및 그래픽: 최소 3:1 비율
- CSS `color`와 `background-color` 조합 분석

**2. 대체 텍스트 (Alternative Text)**
- `<img>` 태그의 `alt` 속성 존재 여부
- 의미 있는 이미지의 `alt` 내용 적절성
- 장식용 이미지의 `alt=""` 처리
- 아이콘 버튼의 `aria-label` 존재 여부
- SVG의 `role="img"` + `<title>` 처리

**3. 키보드 네비게이션 (Keyboard Navigation)**
- 모든 인터랙티브 요소의 키보드 접근 가능 여부
- 논리적인 탭 순서 (`tabIndex` 오남용 확인)
- `tabIndex="-1"` 적절한 사용
- 트랩 포커스(Modal, Dialog) 구현 여부
- ESC 키로 닫기 기능 구현 여부

**4. 포커스 관리 (Focus Management)**
- `:focus` 또는 `:focus-visible` 스타일 정의 여부
- `outline: none` 단독 사용 금지 확인
- 동적 콘텐츠 로드 후 포커스 이동 처리
- 모달/다이얼로그 열기 시 포커스 이동

**5. 스크린리더 지원 (Screen Reader)**
- 시맨틱 HTML 태그 사용 여부 (`button` vs `div onClick`)
- ARIA 랜드마크 역할 (`main`, `nav`, `aside`, `header`, `footer`)
- `aria-live` 영역으로 동적 콘텐츠 알림
- `aria-expanded`, `aria-selected`, `aria-checked` 상태 관리
- `role` 속성 올바른 사용

## 출력 형식

### 검사 결과 요약
| 항목 | 상태 | 위반 수 |
|------|------|--------|
| 명암비 | ✅ / ⚠️ / ❌ | N |
| 대체 텍스트 | ✅ / ⚠️ / ❌ | N |
| 키보드 네비게이션 | ✅ / ⚠️ / ❌ | N |
| 포커스 관리 | ✅ / ⚠️ / ❌ | N |
| 스크린리더 | ✅ / ⚠️ / ❌ | N |

### 위반 상세 목록
각 위반에 대해:
- **위치**: 파일명:줄번호
- **심각도**: Critical / Major / Minor
- **위반 내용**: WCAG 기준 항목 및 설명
- **현재 코드**: (문제 코드 스니펫)
- **개선 방안**: (수정된 코드 스니펫)

### 자동 수정 가능한 항목
수정 코드를 직접 제공하여 즉시 적용 가능한 항목을 나열하라.

### WCAG 준수율
전체 검사 항목 대비 통과 비율을 퍼센트로 표시하라.
